# 验证理论

## 核心定义

验证理论（Verification Theory）定义了在声称工作完成之前，必须运行验证命令并确认输出的方法论框架。

## 核心原则

**证据先于声明，总是如此。**

```
┌─────────────────────────────────────────────────────────────┐
│                    铁律                                     │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   没有新鲜验证证据，就不能声称完成                           │
│                                                              │
│   如果你没有在这条消息中运行验证命令，你就不能声称它通过。   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 门禁函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令证明这个声明？
2. 运行：执行完整命令（新鲜的，完整的）
3. 阅读：完整输出，检查退出代码，计算失败
4. 验证：输出确认声明吗？
   - 如果否：用证据陈述实际状态
   - 如果是：用证据陈述声明
5. 只有那时：做出声明

跳过任何步骤 = 撒谎，不是验证
```

## 常见失败

| 声明 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0失败 | 之前运行，"应该通过" |
| Linter干净 | Linter输出：0错误 | 部分检查，推断 |
| 构建成功 | 构建命令：退出0 | Linter通过，日志看起来好 |
| Bug修复 | 测试原始症状：通过 | 代码改变，假设修复 |
| 回归测试有效 | 红绿循环验证 | 测试通过一次 |
| Agent完成 | VCS diff显示改变 | Agent报告"成功" |
| 需求满足 | 逐行检查清单 | 测试通过 |

## 红旗警告

- 使用"应该"、"可能"、"似乎"
- 在验证之前表达满意（"太好了！"、"完美！"、"完成！"等）
- 没有验证就要提交/推送/PR
- 信任Agent成功报告
- 依赖部分验证
- 认为"就这一次"
- 疲惫想让工作结束
- **任何暗示成功但没有运行验证的措辞**

## 关键模式

### 测试

```
✅ [运行测试命令] [看到：34/34通过] "所有测试通过"
❌ "应该现在通过" / "看起来正确"
```

### 回归测试（TDD红绿）

```
✅ 编写 → 运行（通过）→ 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了回归测试"（没有红绿验证）
```

### 构建

```
✅ [运行构建] [看到：退出0] "构建通过"
❌ "Linter通过"（linter不检查编译）
```

### 需求

```
✅ 重新阅读计划 → 创建检查清单 → 验证每个 → 报告差距或完成
❌ "测试通过，阶段完成"
```

### Agent委托

```
✅ Agent报告成功 → 检查VCS diff → 验证改变 → 报告实际状态
❌ 信任Agent报告
```

## 合理化预防

| 借口 | 现实 |
|------|------|
| "应该现在工作" | 运行验证 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter通过" | Linter ≠ 编译器 |
| "Agent说成功" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查够了" | 部分证明不了什么 |
| "不同的词所以规则不适用" | 精神胜于字面 |

## 为什么这很重要

从24次失败记忆中：
- 用户说"我不相信你" - 信任破裂
- 未定义函数发布 - 会崩溃
- 缺失需求发布 - 不完整功能
- 浪费时间在虚假完成 → 重定向 → 返工
- 违反："诚实是核心价值观。如果你撒谎，你将被替换。"

## 何时应用

**总是在以下之前：**
- 任何成功/完成声明的变体
- 任何满意的表达
- 任何关于工作状态的积极陈述
- 提交、PR创建、任务完成
- 移动到下一个任务
- 委托给Agent

**规则适用于：**
- 确切短语
- 释义和同义词
- 成功的含义
- 任何暗示完成/正确的沟通

## 与其他理论的关系

| 理论 | 关系 |
|------|------|
| **nopua** | 提供内在动机：因为自尊而验证 |
| **TDD理论** | 验证是TDD的核心部分 |
| **系统调试理论** | 修复后需要验证 |

## 适用团队类型

| 团队类型 | 应用方式 |
|---------|---------|
| **development** | 每个完成声明前验证 |
| **testing** | 验证测试结果 |
| **review** | 验证评审发现 |

## 关键要点

1. **运行命令** - 不猜测，不假设
2. **阅读输出** - 完整阅读，检查退出代码
3. **粘贴证据** - 显示证明，不是言语
4. **没有例外** - 疲惫、紧急、简单都不跳过
5. **证据先于声明** - 顺序很重要
