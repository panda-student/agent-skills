# 计划编写理论

## 核心定义

计划编写理论（Writing Plans Theory）定义了为没有上下文的工程师编写全面实现计划的方法论框架。

## 核心原则

**假设工程师对你的代码库几乎没有上下文，品味也有问题。记录他们需要知道的一切。**

```
┌─────────────────────────────────────────────────────────────┐
│                    计划哲学                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   他们是熟练的开发者，但：                                   │
│   • 几乎不了解你的工具集                                     │
│   • 几乎不了解你的问题领域                                   │
│   • 不知道很好的测试设计                                     │
│                                                              │
│   给他们完整的计划作为一口大小的任务                         │
│   DRY. YAGNI. TDD. 频繁提交.                                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 文件结构

在定义任务之前，规划出哪些文件将被创建或修改，每个文件负责什么。

- 设计有清晰边界和明确定义接口的单元
- 每个文件应该有一个明确的职责
- 一起改变的文件应该在一起
- 按职责分割，不是技术层
- 在现有代码库中，遵循既有模式

## 一口大小的任务粒度

**每一步是一个动作（2-5分钟）：**

- "编写失败测试" - 步骤
- "运行以确保失败" - 步骤
- "实现最小代码使测试通过" - 步骤
- "运行测试确保通过" - 步骤
- "提交" - 步骤

## 计划文档头

```markdown
# [功能名称] 实现计划

> **对于代理工作者：** 必需子技能：使用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans 来逐任务实现此计划。步骤使用复选框（`- [ ]`）语法进行跟踪。

**目标：** [描述构建什么的一句话]

**架构：** [关于方法的2-3句话]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试验证失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：FAIL with "function not defined"

- [ ] **步骤 3：编写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试验证通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 禁止占位符

每一步必须包含工程师需要的实际内容。这些是**计划失败**——永远不要写它们：

- "TBD"、"TODO"、"之后实现"、"填写细节"
- "添加适当的错误处理"/ "添加验证"/ "处理边界情况"
- "为上述编写测试"（没有实际测试代码）
- "类似于任务N"（重复代码——工程师可能不按顺序阅读任务）
- 描述做什么而不显示如何的步骤（代码步骤需要代码块）
- 引用任何任务中未定义的类型、函数或方法

## 自审

编写完整计划后，用新眼光看计划并对照规格检查。

**1. 规格覆盖：** 略读规格中的每个部分/需求。你能指出实现它的任务吗？列出任何差距。

**2. 占位符扫描：** 在计划中搜索红旗——上面"禁止占位符"部分的任何模式。修复它们。

**3. 类型一致性：** 你在后续任务中使用的类型、方法签名和属性名称是否与早期任务中定义的匹配？任务3中调用的函数`clearLayers()`但任务7中是`clearFullLayers()`是bug。

如果发现问题，内联修复。不需要重新评审——只是修复并继续。

## 范围检查

如果规格覆盖多个独立子系统，它应该在头脑风暴期间分解为子项目规格。如果没有，建议分解为单独的计划——每个子系统一个。每个计划应该自己产生可工作的、可测试的软件。

## 执行交接

保存计划后，提供执行选择：

```
计划完成并保存到 `docs/superpowers/plans/<filename>.md`。两种执行选项：

1. 子代理驱动（推荐） - 我每个任务派遣一个新子代理，任务间评审，快速迭代

2. 内联执行 - 使用 executing-plans 在此会话中执行任务，批量执行带检查点

哪种方法？
```

## 与其他理论的关系

| 理论 | 关系 |
|------|------|
| **头脑风暴理论** | 设计批准后编写计划 |
| **TDD理论** | 计划中的每一步遵循TDD |
| **子代理驱动开发** | 计划由子代理执行 |
| **验证理论** | 每步都有验证命令 |

## 适用团队类型

| 团队类型 | 应用方式 |
|---------|---------|
| **planning** | 核心技能，将设计转化为计划 |
| **development** | 开发团队遵循计划 |

## 关键要点

1. **总是确切文件路径** - 不模糊
2. **每一步完整代码** - 如果改变代码，显示代码
3. **确切命令和预期输出** - 可验证
4. **DRY, YAGNI, TDD, 频繁提交** - 遵循原则
5. **无占位符** - 每一步都完整
