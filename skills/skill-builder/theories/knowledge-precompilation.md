# 知识预编译原则

## 核心定义

**知识预编译原则**（Knowledge Pre-compilation Principle）：将确定性、可重复的执行逻辑封装为命令、工具或脚本，避免在运行时进行重复推理。

## 问题背景

### 传统模式的痛点

在 AI 辅助开发中，传统模式存在以下问题：

1. **推理开销大**：每次执行相同任务都需要重新推理
2. **结果不稳定**：相同输入可能产生不同输出
3. **上下文浪费**：重复加载相同的知识内容
4. **错误率高**：复杂推理链条增加出错概率

### 预编译的优势

| 维度 | 传统模式 | 预编译模式 |
|------|---------|-----------|
| 执行速度 | 慢（需推理） | 快（直接执行） |
| 结果稳定性 | 不稳定 | 稳定一致 |
| 上下文消耗 | 高 | 低 |
| 错误率 | 较高 | 较低 |
| 可维护性 | 分散 | 集中 |

## 理论基础

### 1. 确定性边界

**确定性逻辑**：给定相同输入，总是产生相同输出的逻辑。
- 数学计算
- 格式转换
- 代码编译
- 规则检查

**非确定性逻辑**：输出可能随上下文变化的逻辑。
- 创意写作
- 方案设计
- 问题诊断
- 决策建议

**原则**：只对确定性逻辑进行预编译封装。

### 2. 频率-确定性矩阵

```
                    高频使用
                       │
         ┌─────────────┼─────────────┐
         │             │             │
    优先封装      优先封装      建议封装
    (脚本/工具)    (npm script)   (文档化)
         │             │             │
    低确定性 ────────┼──────── 高确定性
         │             │             │
    建议封装      建议封装      可选封装
    (交互式工具)   (简单命令)    (手动执行)
         │             │             │
         └─────────────┼─────────────┘
                       │
                    低频使用
```

### 3. 封装层次

```
Layer 0: 原子命令（单行命令）
    └── npm run lint

Layer 1: 组合脚本（多命令组合）
    └── npm run test:all = lint + typecheck + test

Layer 2: 工具封装（带参数化）
    └── npm run gen:component -- --name=Button

Layer 3: 工作流封装（多步骤流程）
    └── npm run release = test + build + publish
```

## 实践指南

### 判断是否需要封装

**需要封装的信号**：
- 同一操作执行超过 3 次
- 操作步骤固定不变
- 执行结果可预测
- 多人需要执行相同操作

**不需要封装的信号**：
- 操作频率很低
- 步骤每次都变化
- 需要大量人工判断
- 一次性任务

### 封装优先级

| 优先级 | 条件 | 封装方式 | 示例 |
|-------|------|---------|------|
| P0 | 高频 + 高确定性 | 独立脚本/工具 | lint、test、build |
| P1 | 高频 + 中确定性 | npm script | gen:component |
| P2 | 中频 + 高确定性 | shell 脚本 | deploy.sh |
| P3 | 中频 + 中确定性 | 文档化命令 | README 中的命令序列 |
| P4 | 低频或低确定性 | 可选封装 | 保持灵活性 |

### 封装原则

1. **单一职责**：每个脚本只做一件事
   ```bash
   # 好：职责单一
   npm run lint
   npm run test
   
   # 差：职责混杂
   npm run lint-and-test-and-build
   ```

2. **幂等性**：重复执行产生相同结果
   ```bash
   # 好：幂等设计
   npm run db:migrate  # 检查已执行则跳过
   
   # 差：非幂等
   npm run db:seed     # 每次都插入重复数据
   ```

3. **可测试**：封装后必须可验证
   ```bash
   # 好：有验证步骤
   npm run build && npm run test
   
   # 差：无验证
   npm run build
   ```

4. **可配置**：支持参数化输入
   ```bash
   # 好：支持参数
   npm run gen:component -- --name=Button --type=react
   
   # 差：硬编码
   npm run gen:button-component
   ```

5. **有文档**：包含使用说明
   ```bash
   # 好：有帮助信息
   ./deploy.sh --help
   
   # 差：无文档
   ./deploy.sh  # 用户不知道参数含义
   ```

## 反模式警示

### 反模式 1：过度封装

```bash
# 错误：将非确定性逻辑封装
npm run suggest-architecture  # 架构建议需要推理

# 正确：保持灵活性
# 由 AI 根据上下文提供架构建议
```

### 反模式 2：封装粒度过粗

```bash
# 错误：一个脚本做太多事
npm run deploy-all  # 包含 build、test、deploy、notify

# 正确：拆分为独立步骤
npm run build
npm run test
npm run deploy
```

### 反模式 3：缺乏错误处理

```bash
# 错误：无错误处理
npm run build
npm run deploy  # build 失败也会执行

# 正确：有错误处理
npm run build && npm run deploy
```

### 反模式 4：硬编码路径

```bash
# 错误：硬编码
cp ./src/components/Button.tsx ./dist/

# 正确：使用变量
cp "${SRC_DIR}/components/${COMPONENT}.tsx" "${DIST_DIR}/"
```

## 与其他原则的关系

### 与渐进式披露的关系

- **渐进式披露**：解决知识加载问题
- **知识预编译**：解决知识执行问题
- **结合**：先按需加载，再预编译执行

### 与单一职责的关系

- **单一职责**：每个文件职责单一
- **知识预编译**：每个脚本职责单一
- **结合**：封装粒度遵循单一职责

## 实践案例

### 案例 1：代码质量检查

**问题**：每次都需要手动执行多个检查命令

**解决方案**：
```json
{
  "scripts": {
    "lint": "eslint src/",
    "typecheck": "tsc --noEmit",
    "test": "jest",
    "test:all": "npm run lint && npm run typecheck && npm run test"
  }
}
```

**收益**：
- 执行时间：从 5 分钟推理 → 30 秒执行
- 稳定性：结果一致可预测
- 可维护性：集中管理检查规则

### 案例 2：组件生成

**问题**：每次创建组件需要手动创建多个文件

**解决方案**：
```bash
# scripts/gen-component.sh
#!/bin/bash
NAME=$1
mkdir -p "src/components/${NAME}"
cat > "src/components/${NAME}/index.ts" << EOF
export { ${NAME} } from './${NAME}';
EOF
# ... 更多模板
```

**收益**：
- 执行时间：从 10 分钟 → 10 秒
- 一致性：所有组件结构相同
- 可维护性：修改模板即可更新所有组件

## 总结

知识预编译原则是提高 AI 辅助开发效率的关键策略：

1. **识别确定性逻辑**：找到可以预编译的操作
2. **评估封装优先级**：基于频率和确定性决策
3. **遵循封装原则**：单一职责、幂等、可测试
4. **避免反模式**：不过度、不粗粒度、有错误处理

通过知识预编译，可以将 AI 的能力聚焦于需要推理和创造的任务，而将确定性操作交给预编译的脚本执行，实现人机协作的最佳效果。
