# 头脑风暴理论

## 核心定义

头脑风暴理论（Brainstorming Theory）定义了在实现任何创意工作之前，将想法转化为完整设计和规格的方法论框架。

## 核心原则

**设计先行，防止未思考就编码**

```
┌─────────────────────────────────────────────────────────────┐
│                    设计优先原则                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ❌ 错误：直接编码                                          │
│   用户请求 → 立即实现 → 发现问题 → 重做                      │
│                                                              │
│   ✅ 正确：设计先行                                          │
│   用户请求 → 理解需求 → 设计方案 → 获得批准 → 实现           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 核心流程

### 1. 探索项目上下文

在开始询问问题之前，首先了解：
- 当前文件结构
- 项目文档
- 最近提交
- 现有模式

### 2. 一次一个问题

- 每条消息只问一个问题
- 优先使用多选题
- 开放式问题也可以
- 专注于理解：目的、约束、成功标准

### 3. 提出2-3个方案

- 每个方案都有权衡分析
- 给出推荐意见和理由
- 以推荐选项开头并解释原因

### 4. 展示设计

- 每个部分的详细程度与其复杂性相匹配
- 简单的几句话，复杂的200-300字
- 每个部分后询问是否正确
- 覆盖：架构、组件、数据流、错误处理、测试

### 5. 获取批准

**硬性规则**：在展示设计并获得用户批准之前，不要调用任何实现技能。

## 设计隔离原则

将系统分解为更小的单元：
- 每个单元有一个明确的目的
- 通过明确定义的接口通信
- 可以独立理解和测试

**检验标准**：
- 这个单元做什么？
- 如何使用它？
- 它依赖什么？

## 反模式

### "这太简单了不需要设计"

每个项目都要经历这个过程。待办列表、单一功能工具、配置更改——所有这些。"简单"项目正是未审查假设导致最多浪费工作的地方。

设计可以很短（对于真正简单的项目只需几句话），但你必须展示它并获得批准。

## 在现有代码库中工作

- 在提出更改之前探索当前结构
- 遵循现有模式
- 如果现有代码有问题影响工作，包含有针对性的改进
- 不要提出无关的重构

## 与其他理论的关系

| 理论 | 关系 |
|------|------|
| **nopua** | 提供内在动机：因为想做好而设计，不是害怕惩罚 |
| **验证理论** | 设计完成后需要验证 |
| **计划编写理论** | 设计批准后编写实现计划 |

## 适用团队类型

| 团队类型 | 应用方式 |
|---------|---------|
| **planning** | 核心技能，用于需求分析和架构设计 |
| **development** | 开发前理解需求 |
| **review** | 评审设计文档 |

## 关键要点

1. **一次一个问题** - 不要用多个问题淹没用户
2. **多选题优先** - 比开放问题更容易回答
3. **YAGNI无情** - 从所有设计中删除不必要的功能
4. **探索替代方案** - 在确定之前总是提出2-3个方案
5. **增量验证** - 展示设计，在继续之前获得批准
