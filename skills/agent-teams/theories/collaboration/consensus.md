# 团队共识理论

## 一、AI 决策原则

**所有业务决策由主 Agent（AI）完成，引擎只负责状态管理。**

```
┌──────────────────┐     ┌──────────────────┐
│   主 Agent (AI)   │     │    Engine        │
│                  │     │                  │
│ ✓ 分析任务复杂度  │     │ ✓ 记录团队状态    │
│ ✓ 决定团队类型    │──▶  │ ✓ 追踪文档交付物  │
│ ✓ 决定成员数量    │     │ ✓ 记录投票决策    │
│ ✓ 决定委托策略    │     │ ✓ 管理委托链      │
└──────────────────┘     └──────────────────┘
```

---

## 二、投票决策机制

### 成员数量约束

**团队成员数必须是奇数，确保投票不会平票。**

```
推荐配置:
- 小型团队: 3 人
- 中型团队: 5 人
- 大型团队: 7 人
```

### 投票类型

| 投票 | 含义 | 使用场景 |
|------|------|---------|
| `approve` | 通过 | 认可结果 |
| `reject` | 反对 | 发现问题 |
| `abstain` | 弃权 | 不确定/中立 |

### 判定规则：一票否决

**规则：只要有1票反对，整体就不通过。**

```
示例（3人团队）:
- 3票通过, 0票反对 → ✅ 通过
- 2票通过, 1票反对 → ❌ 不通过（一票否决）
- 1票通过, 2票反对 → ❌ 不通过

示例（5人团队）:
- 5票通过, 0票反对 → ✅ 通过
- 4票通过, 1票反对 → ❌ 不通过（一票否决）
```

**设计原因**：
- 质量问题不能妥协
- 任何一个成员发现的严重问题都应阻止发布
- 确保每个成员都有实际的决定权

## 三、文档交付物

### 记录原则

**只记录文档类交付物，不记录代码文件。**

### 识别类型

| 后缀 | 类型 |
|------|------|
| `.md` | Markdown 文档 |
| `.doc/.docx` | Word 文档 |
| `.pdf` | PDF 文档 |
| `.txt` | 纯文本 |
| `.rst` | reStructuredText |

### 各团队典型交付物

| 团队类型 | 典型文档交付物 |
|---------|---------------|
| planning | 需求文档、架构设计文档、API规范 |
| development | 无（代码不记录） |
| testing | 测试计划、测试报告 |
| review | 评审报告、问题清单 |

---

## 四、决策职责

| 决策类型 | 决策者 | 依据 |
|---------|--------|------|
| 团队类型 | 主 Agent | 任务性质、技术领域 |
| 成员数量 | 主 Agent | 任务复杂度（奇数） |
| 投票结果 | 团队成员 | 专业判断 |
| 是否委托 | 主 Agent | 投票结果、任务状态 |

---

## 五、委托循环

```
团队A 执行 → 成员投票 → 记录结果
    │
    ├── 全部通过 → COMPLETE → 结束
    │
    └── 存在反对 → DELEGATE → 委托给其他团队
                              │
                              ▼
                         创建团队B...
```
