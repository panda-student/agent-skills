# 子代理驱动开发理论

## 核心定义

子代理驱动开发理论（Subagent-Driven Development Theory）定义了通过为每个任务派遣新的子代理，并在每个任务后进行两阶段评审来实现计划的方法论框架。

## 核心原则

**每个任务新子代理 + 两阶段评审（规格然后质量）= 高质量，快速迭代**

```
┌─────────────────────────────────────────────────────────────┐
│                    核心架构                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                 主 Agent（调度器）                    │   │
│   │  • 提取所有任务和上下文                               │   │
│   │  • 派遣实现子代理                                     │   │
│   │  • 派遣评审子代理                                     │   │
│   │  • 汇总和报告                                        │   │
│   └─────────────────────────────────────────────────────┘   │
│                           │                                  │
│                           ▼                                  │
│   ┌─────────────────────────────────────────────────────┐   │
│   │              子代理（独立上下文）                     │   │
│   │  每个任务获得全新的 ~200k token 上下文               │   │
│   │  不继承主 Agent 的会话历史                            │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 为什么使用子代理

你将任务委托给具有隔离上下文的专门代理。通过精确构建他们的指令和上下文，你确保他们保持专注并成功完成任务。他们不应该继承你的会话上下文或历史——你构建他们需要的东西。这也保留了你自己的上下文用于协调工作。

## 流程

### 每任务流程

```
┌─────────────────────────────────────────────────────────────┐
│                    任务执行流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 派遣实现子代理                                          │
│     │                                                        │
│     ├── 子代理有问题？→ 回答问题，提供上下文                 │
│     │                                                        │
│     └── 子代理实现、测试、提交、自审                         │
│                                                              │
│  2. 派遣规格评审子代理                                       │
│     │                                                        │
│     ├── 不符合规格？→ 实现子代理修复 → 重新评审              │
│     │                                                        │
│     └── 符合规格？→ 继续                                     │
│                                                              │
│  3. 派遣代码质量评审子代理                                   │
│     │                                                        │
│     ├── 有问题？→ 实现子代理修复 → 重新评审                  │
│     │                                                        │
│     └── 批准？→ 标记任务完成                                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 两阶段评审

### 第一阶段：规格合规评审

**目的**：确保代码完全匹配规格，没有多余或缺失。

检查：
- 所有需求都实现了吗？
- 有没有实现未要求的东西？
- 规格中的每个点都有对应代码吗？

### 第二阶段：代码质量评审

**目的**：确保实现质量高。

检查：
- 代码清晰可读？
- 有适当的错误处理？
- 有测试？
- 遵循项目约定？

**关键**：规格评审必须在代码质量评审之前。

## 模型选择

使用能够处理每个角色的最不强力的模型，以节省成本并提高速度。

| 任务类型 | 推荐模型 |
|---------|---------|
| 机械实现任务（隔离函数，清晰规格，1-2个文件） | 快速、便宜的模型 |
| 集成和判断任务（多文件协调，模式匹配，调试） | 标准模型 |
| 架构、设计和评审任务 | 最强大的可用模型 |

## 处理实现者状态

| 状态 | 含义 | 处理 |
|------|------|------|
| **DONE** | 完成，准备评审 | 继续规格合规评审 |
| **DONE_WITH_CONCERNS** | 完成但有疑虑 | 阅读担忧后再继续 |
| **NEEDS_CONTEXT** | 需要更多信息 | 提供缺失上下文，重新派遣 |
| **BLOCKED** | 无法完成 | 评估阻塞：提供上下文？更强模型？分解任务？升级？ |

## 红旗警告

**绝不：**
- 在main/master分支上开始实现（没有明确用户同意）
- 跳过评审（规格合规或代码质量）
- 有未修复问题就继续
- 并行派遣多个实现子代理（冲突）
- 让子代理读取计划文件（提供完整文本）
- 跳过场景设置上下文
- 忽略子代理问题
- 在规格合规上接受"足够接近"
- 跳过评审循环
- 让实现者自审替代实际评审
- **在规格合规✅之前开始代码质量评审**（错误顺序）
- 有开放评审问题时移动到下一个任务

## 代理提示结构

好的代理提示：
1. **聚焦** - 一个清晰的问题域
2. **自包含** - 理解问题所需的所有上下文
3. **输出明确** - 代理应该返回什么？

```markdown
修复src/agents/agent-tool-abort.test.ts中的3个失败测试：

1. "应该用部分输出捕获中止工具" - 期望消息中有'interrupted at'
2. "应该处理混合完成和中止的工具" - 快速工具中止而不是完成
3. "应该正确跟踪pendingToolCount" - 期望3个结果但得到0

这些是时序/竞争条件问题。你的任务：

1. 阅读测试文件并理解每个测试验证什么
2. 识别根本原因 - 时序问题还是实际bug？
3. 通过以下方式修复：
   - 用基于事件的等待替换任意超时
   - 如果发现，修复中止实现中的bug
   - 如果测试改变了行为，调整测试期望

不要只是增加超时 - 找到真正的问题。

返回：你发现的和修复的摘要。
```

## 常见错误

| 错误 | 问题 | 修复 |
|------|------|------|
| 太宽泛 | "修复所有测试" - 代理迷失 | "修复agent-tool-abort.test.ts" - 聚焦范围 |
| 没上下文 | "修复竞争条件" - 代理不知道在哪里 | 粘贴错误消息和测试名称 |
| 没约束 | 代理可能重构一切 | "不要改变生产代码"或"只修复测试" |
| 输出模糊 | "修复它" - 你不知道改变了什么 | "返回根本原因和改变的摘要" |

## 与其他理论的关系

| 理论 | 关系 |
|------|------|
| **nopua** | 提供内在动机：因为想做对而评审 |
| **并行执行理论** | 子代理可以并行执行独立任务 |
| **TDD理论** | 子代理应该使用TDD |
| **验证理论** | 评审是验证的形式 |

## 适用团队类型

| 团队类型 | 应用方式 |
|---------|---------|
| **development** | 核心技能，执行实现计划 |
| **review** | 评审子代理用于质量保证 |
| **testing** | 测试验证 |

## 关键要点

1. **每个任务新子代理** - 不继承上下文，防止污染
2. **两阶段评审** - 先规格合规，后代码质量
3. **评审循环** - 发现问题 → 修复 → 重新评审
4. **明确提示** - 聚焦、自包含、输出明确
5. **模型选择** - 用足够的最便宜模型
