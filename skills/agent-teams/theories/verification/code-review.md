# 代码评审理论

## 核心定义

代码评审理论（Code Review Theory）定义了如何正确请求和接收代码评审的方法论框架。

## 第一部分：请求代码评审

### 核心原则

**尽早评审，经常评审。**

### 何时请求评审

**强制性：**
- 子代理驱动开发中的每个任务后
- 完成主要功能后
- 合并到main之前

**可选但有价值：**
- 卡住时（新视角）
- 重构前（基线检查）
- 修复复杂bug后

### 如何请求

1. **获取git SHA：**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

2. **派遣代码评审子代理：**
使用带有代码评审模板的Task工具

3. **对反馈采取行动：**
- 立即修复关键问题
- 在继续之前修复重要问题
- 稍后注意次要问题
- 如果评审者错误则反驳（带理由）

### 示例

```
[刚完成任务2：添加验证函数]

你：让我在继续之前请求代码评审。

BASE_SHA=$(git log --oneline | grep "任务 1" | head -1 | awk '{print $1}')
HEAD_SHA=$(git rev-parse HEAD)

[派遣代码评审子代理]
  WHAT_WAS_IMPLEMENTED: 对话索引的验证和修复函数
  PLAN_OR_REQUIREMENTS: docs/superpowers/plans/deployment-plan.md中的任务2
  BASE_SHA: a7981ec
  HEAD_SHA: 3df7661
  DESCRIPTION: 添加了verifyIndex()和repairIndex()，有4种问题类型

[子代理返回]：
  优点：清晰的架构，真实测试
  问题：
    重要：缺少进度指示器
    次要：报告间隔的魔法数字（100）
  评估：准备继续

你：[修复进度指示器]
[继续任务3]
```

## 第二部分：接收代码评审

### 核心原则

**实现前验证。假设前询问。技术正确性优于社交舒适。**

### 响应模式

```
当收到代码评审反馈时：

1. 阅读：完整反馈，不反应
2. 理解：用自己的话重述需求（或询问）
3. 验证：对照代码库现实检查
4. 评估：对这个代码库技术上合理吗？
5. 响应：技术确认或有理由的反驳
6. 实现：一次一项，每项测试
```

### 禁止响应

**绝不：**
- "你完全正确！"（明确的CLAUDE.md违规）
- "好观点！" / "优秀反馈！"（表演性）
- "让我现在实现"（验证前）

**代替：**
- 重述技术需求
- 问澄清问题
- 如果错误则用技术理由反驳
- 直接开始工作（行动 > 言语）

### 处理不清晰反馈

```
如果有任何项目不清晰：
  停止 - 暂不实现任何东西
  询问不清晰项目的澄清

为什么：项目可能相关。部分理解 = 错误实现。
```

**示例：**
```
用户："修复1-6"
你理解1,2,3,6。不清楚4,5。

❌ 错误：现在实现1,2,3,6，稍后问4,5
✅ 正确："我理解项目1,2,3,6。在继续之前需要4和5的澄清。"
```

### 来源特定处理

#### 来自用户
- **信任** - 理解后实现
- **如果范围不清楚仍要询问**
- **没有表演性同意**
- **跳到行动**或技术确认

#### 来自外部评审者
```
实现前：
  1. 检查：对这个代码库技术正确吗？
  2. 检查：破坏现有功能吗？
  3. 检查：当前实现的原因？
  4. 检查：在所有平台/版本上工作吗？
  5. 检查：评审者理解完整上下文吗？

如果建议似乎错误：
  用技术理由反驳

如果无法轻易验证：
  说出来："我无法在没有[X]的情况下验证这个。我应该[调查/询问/继续]？"

如果与用户之前的决定冲突：
  停止并先与用户讨论
```

### YAGNI检查

```
如果评审者建议"正确实现"：
  在代码库中grep实际使用

  如果未使用："这个端点没被调用。删除它（YAGNI）？"
  如果使用：然后正确实现
```

### 实现顺序

```
对于多项目反馈：
  1. 首先澄清任何不清楚的项目
  2. 然后按此顺序实现：
     - 阻塞问题（中断、安全）
     - 简单修复（拼写错误、导入）
     - 复杂修复（重构、逻辑）
  3. 单独测试每个修复
  4. 验证没有回归
```

### 何时反驳

当以下情况时反驳：
- 建议破坏现有功能
- 评审者缺乏完整上下文
- 违反YAGNI（未使用的功能）
- 对此技术栈技术上不正确
- 存在遗留/兼容性原因
- 与架构决定冲突

**如何反驳：**
- 使用技术理由，不是防御性
- 问具体问题
- 引用工作的测试/代码
- 如果是架构问题，让用户参与

### 确认正确反馈

当反馈正确时：
```
✅ "已修复。[改变的简要描述]"
✅ "好发现 - [具体问题]。已在[位置]修复。"
✅ [直接修复并在代码中显示]

❌ "你完全正确！"
❌ "好观点！"
❌ "谢谢发现那个！"
❌ "谢谢[任何东西]"
❌ 任何感激表达
```

**为什么不感谢：** 行动说话。直接修复。代码本身显示你听到了反馈。

### 优雅纠正你的反驳

如果你反驳但错了：
```
✅ "你是对的 - 我检查了[X]它确实[Y]。现在实现。"
✅ "验证了这个你是正确的。我最初的理解错了因为[原因]。修复中。"

❌ 长道歉
❌ 为为什么反驳辩护
❌ 过度解释
```

事实性陈述纠正并继续。

## 常见错误

| 错误 | 修复 |
|------|------|
| 表演性同意 | 陈述需求或直接行动 |
| 盲目实现 | 先对照代码库验证 |
| 不测试就批量 | 一次一个，每个测试 |
| 假设评审者正确 | 检查是否破坏东西 |
| 避免反驳 | 技术正确性 > 舒适 |
| 部分实现 | 先澄清所有项目 |
| 无法验证就继续 | 陈述限制，询问方向 |

## 与其他理论的关系

| 理论 | 关系 |
|------|------|
| **nopua** | 提供内在动机：因为想改进而评审 |
| **验证理论** | 评审是验证的形式 |
| **子代理驱动开发** | 评审是核心部分 |

## 适用团队类型

| 团队类型 | 应用方式 |
|---------|---------|
| **review** | 核心技能，执行评审 |
| **development** | 请求和接收评审 |
| **testing** | 验证评审发现 |

## 关键要点

1. **尽早评审，经常评审** - 不要等大PR
2. **技术严谨** - 不是情感表演
3. **先验证** - 不假设正确
4. **愿意反驳** - 带技术理由
5. **澄清不清楚项目** - 不猜测
