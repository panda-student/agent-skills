# 测试驱动开发理论

## 核心定义

测试驱动开发理论（Test-Driven Development Theory）定义了在编写实现代码之前先编写测试的方法论框架。

## 核心原则

**如果你没有看到测试失败，你就不知道它是否测试了正确的东西。**

```
┌─────────────────────────────────────────────────────────────┐
│                    铁律                                     │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   没有失败的测试，就不能有生产代码                           │
│                                                              │
│   先写代码再写测试？删掉它。重新开始。                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 红绿重构循环

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│     红色     │     │     绿色     │     │    重构      │
│  编写失败测试 │ ──▶ │  编写最小代码 │ ──▶ │    清理      │
└──────────────┘     └──────────────┘     └──────────────┘
       │                    │                    │
       ▼                    ▼                    ▼
   验证失败             验证通过             保持通过
   正确原因             所有测试通过          重构
       │                    │                    │
       └────────────────────┴────────────────────┘
                           │
                           ▼
                       下一个测试
```

### 红色阶段 - 编写失败测试

**要求**：
- 一个行为
- 清晰的名称
- 真实代码（除非不可避免，否则不用模拟）

**示例**：
```typescript
test('重试失败操作3次', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('fail');
    return 'success';
  };

  const result = await retryOperation(operation);

  expect(result).toBe('success');
  expect(attempts).toBe(3);
});
```

### 验证红色 - 观察失败

**强制性。绝不跳过。**

确认：
- 测试失败（不是错误）
- 失败消息是预期的
- 失败是因为功能缺失（不是拼写错误）

### 绿色阶段 - 最小代码

编写最简单的代码来通过测试。不要添加功能、重构其他代码或"改进"超出测试范围的内容。

### 验证绿色 - 观察通过

**强制性。**

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（无错误、警告）

### 重构阶段 - 清理

只有在绿色之后：
- 删除重复
- 改进命名
- 提取辅助函数

保持测试通过。不要添加行为。

## 为什么顺序很重要

### "我会在之后写测试来验证它工作"

之后编写的测试会立即通过。立即通过证明不了什么：
- 可能测试错误的东西
- 可能测试实现，不是行为
- 可能错过你忘记的边界情况
- 你从未看到它捕获bug

测试先行强制你看到测试失败，证明它确实测试了某些东西。

### "删除X小时的工作是浪费"

沉没成本谬误。时间已经过去了。你现在的选择：
- 删除并用TDD重写（X更多小时，高信心）
- 保留它并在之后添加测试（30分钟，低信心，可能有bug）

保留无法信任的代码才是浪费。没有真正测试的工作代码是技术债务。

## 常见合理化

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会出错。测试只需30秒。 |
| "我会在之后测试" | 立即通过的测试证明不了什么。 |
| "之后的测试达到相同目标" | 之后测试 = "这做什么？" 测试先行 = "这应该做什么？" |
| "已经手动测试了" | 临时的 ≠ 系统的。没有记录，无法重新运行。 |
| "删除X小时是浪费" | 沉没成本谬误。保留未验证的代码是技术债务。 |
| "作为参考保留，先写测试" | 你会改编它。那是之后测试。删除意味着删除。 |
| "需要先探索" | 可以。扔掉探索，用TDD开始。 |
| "测试困难 = 设计不清楚" | 听从测试。难测试 = 难使用。 |

## 红旗警告

如果你发现自己在想：
- 代码在测试之前
- 实现后测试
- 测试立即通过
- 无法解释为什么测试失败
- "之后"添加测试
- 合理化"就这一次"
- "我已经手动测试了"
- "之后的测试达到相同目的"
- "这是精神而不是仪式"
- "作为参考保留"或"改编现有代码"
- "已经花了X小时，删除是浪费"

**所有这些都意味着：删除代码。用TDD重新开始。**

## 测试反模式

### 反模式1：测试模拟行为

**违规**：
```typescript
// ❌ 错误：测试模拟存在
test('渲染侧边栏', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**修复**：
```typescript
// ✅ 正确：测试真实组件或不模拟
test('渲染侧边栏', () => {
  render(<Page />);  // 不模拟侧边栏
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});
```

### 反模式2：生产类中的测试专用方法

**违规**：
```typescript
// ❌ 错误：destroy()只在测试中使用
class Session {
  async destroy() {  // 看起来像生产API！
    await this._workspaceManager?.destroyWorkspace(this.id);
  }
}
```

**修复**：
```typescript
// ✅ 正确：测试工具处理测试清理
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}
```

### 反模式3：不理解就模拟

**违规**：
```typescript
// ❌ 错误：模拟破坏测试逻辑
test('检测重复服务器', () => {
  // 模拟阻止测试依赖的配置写入！
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 应该抛出 - 但不会！
});
```

## 与其他理论的关系

| 理论 | 关系 |
|------|------|
| **nopua** | 提供内在动机：因为想做好而测试，不是害怕惩罚 |
| **验证理论** | TDD是验证的具体实践 |
| **系统调试理论** | Bug需要失败测试来证明修复 |

## 适用团队类型

| 团队类型 | 应用方式 |
|---------|---------|
| **development** | 核心技能，每个功能都用TDD |
| **testing** | 验证测试质量 |
| **review** | 检查是否有测试 |

## 关键要点

1. **先写测试** - 强制你思考应该做什么
2. **观察失败** - 证明测试确实测试了某些东西
3. **最小代码** - 只写足够的代码通过
4. **重构** - 在绿色之后清理
5. **绝不跳过** - 没有例外
