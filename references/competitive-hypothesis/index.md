# 竞争假设模式

适用于修复bug类任务，通过多个Teammates并行验证不同的根因假设，快速定位问题根源。

---

## 完整工作流程

```
┌──────────────────┐    确认根因    ┌──────────────────┐    验证通过    ┌──────────────────┐
│   阶段一         │ ────────────→ │   阶段二         │ ────────────→ │   阶段三         │
│   根因分析       │               │   修复实施       │               │   验证完成       │
│   (竞争假设)     │               │                  │               │                  │
└──────────────────┘               └──────────────────┘               └──────────────────┘
         │                                │                                │
         │ 输出:                          │ 输出:                          │ 输出:
         │ 根因结论                       │ 修复代码                        │ 验证结果
         │ 验证报告                       │ 测试用例                        │ 问题解决确认
         │                                │                                │
         └────────────────────────────────┴────────────────────────────────┘
                                      │
                                      ▼
                             ┌──────────────┐
                             │ 验证结果？   │
                             └──────────────┘
                               │         │
                         通过  │         │ 问题仍存在
                               ▼         ▼
                        ┌──────────┐  返回阶段一
                        │ 交付完成 │  (新假设)
                        └──────────┘
```

---

## 阶段一：根因分析

### Lead提示词

你是Agent Teams的Lead，负责协调多个Teammates对一个复杂bug进行根因分析。

任务：
- 分析bug描述和相关代码
- 分配3个Teammates分别验证不同的根因假设
- 协调Teammates之间的讨论和挑战
- 综合所有假设，得出最终的根因结论
- 制定修复方案并验证

Bug描述：[此处填写具体bug描述]
相关代码：[此处提供相关代码]

### 假设分配策略

为每个Teammate分配不同类型的假设：
- **Teammate 1**: 前端假设（UI渲染、用户交互、状态管理相关）
- **Teammate 2**: 后端假设（API接口、服务逻辑、业务流程相关）
- **Teammate 3**: 数据假设（数据存储、数据传输、数据一致性相关）
- **Teammate 4**: 代码假设（代码逻辑、算法实现、边界条件相关）
- **Teammate 5**: 整体业务假设（业务流程、系统集成、依赖关系相关）

### 交付物要求

每个Teammate必须提供：
1. 假设验证报告（包含验证过程、证据、结论）
2. 相关代码片段分析
3. 可复现的测试用例（如果适用）
4. 与其他假设的对比分析

### 输出格式

最终报告应包含：
- 问题概述
- 所有假设的验证结果
- 根因分析结论
- 修复建议
- 预防措施

---

### Teammate提示词

你是Agent Teams的Teammate，负责验证一个具体的bug根因假设。

假设：[此处填写具体假设]
假设类型：[前端/后端/数据/代码/整体业务]
Bug描述：[此处填写具体bug描述]
相关代码：[此处提供相关代码]

### 任务清单

1. **假设分析**: 分析假设的合理性，识别关键条件
2. **代码验证**: 通过代码分析验证假设，检查相关代码路径
3. **证据收集**: 收集支持或反驳假设的证据，记录关键代码片段
4. **测试验证**: 设计测试用例验证假设，执行测试并记录结果
5. **报告撰写**: 提供详细的验证报告，说明验证过程和结果
6. **协作交流**: 与其他Teammates交流发现，回应其他假设的挑战

### 验证方法

- **静态分析**: 审查代码逻辑和数据流
- **动态分析**: 运行代码并观察行为
- **日志分析**: 检查运行时日志和错误信息
- **对比分析**: 与正常工作的代码进行对比
- **边界测试**: 测试边界条件和异常情况

### 报告格式

```markdown
# 假设验证报告

## 假设描述
[假设的详细描述]

## 验证过程
1. [步骤1]
2. [步骤2]
...

## 关键发现
- [发现1]
- [发现2]
...

## 证据
- [代码片段1]
- [日志信息1]
- [测试结果1]

## 结论
- 假设成立/不成立
- 置信度：[高/中/低]
- 理由：[详细说明]

## 与其他假设的关系
- [说明与其他假设的关联或矛盾]
```

### 时间要求

请在[X]小时内完成验证并提交报告。

### 质量标准

- 验证过程清晰可追溯
- 证据充分且相关
- 结论明确且有依据
- 报告结构完整且易读

## 🔄 阶段转换：根因分析 → 修复实施

当根因确认后（至少一个假设成立且置信度高），Lead 必须执行以下转换流程：

### 转换检查清单

- [ ] 至少一个假设已验证成立
- [ ] 根因置信度高（有充分证据支持）
- [ ] 其他假设已被排除或优先级较低
- [ ] 修复方案已初步制定

### 转换输出

```
🔍 根因分析完成

✅ 根因确认：
- 问题根源：[根因描述]
- 置信度：高
- 支持证据：
  - [证据1]
  - [证据2]

📊 假设验证结果：
- 假设1：❌ 不成立
- 假设2：✅ 成立（根因）
- 假设3：⚠️ 部分相关

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔧 正在启动修复实施...

1. 制定修复方案
2. 分配修复任务
3. 实施修复

👉 进入修复实施阶段
```

## 阶段二：修复实施

### Lead提示词

你是Agent Teams的Lead，负责根据根因分析结果实施修复。

任务：
- 根据确认的根因制定修复方案
- 分配修复任务给Teammates
- 协调修复进度
- 确保修复不引入新问题

根因：[已确认的根因]
修复方案：[初步修复方案]

### 修复流程

1. **方案确认**: 最终确认修复方案，评估影响范围
2. **任务分配**: 将修复任务分配给Teammates
3. **代码修复**: 实施代码修改
4. **测试验证**: 编写测试用例验证修复
5. **自测通过**: 确保修复有效且不引入新问题

### 输出格式

```markdown
# 修复实施报告

## 修复概述
- 根因：[根因描述]
- 修复方案：[方案概述]
- 修复时间：[时间]

## 修复内容
### 文件1：[文件名]
- 修改内容：[描述]
- 原因：[说明]

### 文件2：[文件名]
- 修改内容：[描述]
- 原因：[说明]

## 测试验证
- 新增测试用例：[X]个
- 测试结果：✅ 全部通过
- 回归测试：✅ 无影响

## 风险评估
- 影响范围：[描述]
- 潜在风险：[风险及缓解措施]
```

### Teammate提示词

你是Agent Teams的Teammate，负责实施修复。

修复任务：[具体修复任务]
相关文件：[需要修改的文件]
根因：[已确认的根因]

### 任务清单

1. **理解修复方案**: 确保完全理解修复方案和预期效果
2. **代码修改**: 按照方案实施代码修改
3. **测试用例**: 编写测试用例验证修复
4. **自测验证**: 确保修复有效
5. **文档更新**: 更新相关文档

## 🔄 阶段转换：修复实施 → 验证

修复完成后，Lead 必须执行以下转换流程：

### 转换输出

```
🔧 修复实施完成

✅ 已完成：
- 代码修改
- 测试用例编写
- 自测通过

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✅ 正在验证修复...

1. 运行测试用例
2. 验证问题已解决
3. 检查是否有新问题

👉 进入验证阶段
```

---

## 阶段三：验证

### Lead提示词

你是Agent Teams的Lead，负责验证修复是否有效。

任务：
- 运行测试用例验证修复
- 确认原问题已解决
- 检查是否引入新问题
- 如果问题仍存在，返回根因分析

### 验证流程

1. **功能验证**: 验证原问题是否已解决
2. **回归测试**: 确保修复不引入新问题
3. **边界测试**: 测试边界条件
4. **性能验证**: 检查性能是否受影响

### 验证结果判定

| 结果 | 下一步 |
|------|--------|
| 问题已解决，无新问题 | ✅ 交付完成 |
| 问题已解决，有新问题 | 返回修复实施 |
| 问题未解决 | 返回根因分析（新假设） |

### 输出格式

```markdown
# 验证报告

## 验证概述
- 修复内容：[修复概述]
- 验证时间：[时间]

## 验证结果
### 功能验证
- 原问题：✅ 已解决 / ❌ 仍存在
- 复现步骤：[步骤]
- 结果：[描述]

### 回归测试
- 测试用例：[X]个
- 通过：[X]个
- 失败：[X]个

### 边界测试
- 测试场景：[列表]
- 结果：✅ 全部通过

## 结论
- 验证结果：[通过/不通过]
- 问题状态：[已解决/仍存在]
- 下一步：[行动]
```

### ✅ 验证通过 - 交付完成

```
🎉 验证完成 - 问题已解决！

✅ 验证结果：
- 原问题：已解决
- 回归测试：通过
- 边界测试：通过
- 性能影响：无

📦 交付清单：
- [ ] 代码已提交
- [ ] 测试已通过
- [ ] 文档已更新
- [ ] 部署已完成

👉 Bug 修复完成！
```

### ❌ 验证不通过 - 返回根因分析

```
❌ 验证完成 - 问题仍存在

📊 验证结果：
- 原问题：仍存在
- 可能原因：根因分析不完整或有遗漏

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔄 返回根因分析阶段...

需要提出新的假设或重新评估现有假设。

👉 返回根因分析
```

### 自动触发规则

验证完成后，Lead 必须根据结果决定下一步：

1. **问题已解决** → 交付完成，任务结束
2. **问题仍存在** → 返回阶段一，提出新假设重新分析
3. **引入新问题** → 返回阶段二，修复新问题

## 完整流程示例

```
用户：修复一个登录失败的bug

Lead：开始根因分析，分配假设：
- Teammate 1: 前端假设（请求未正确发送）
- Teammate 2: 后端假设（认证逻辑问题）
- Teammate 3: 数据假设（用户数据问题）

Teammates 并行验证...

Lead：汇总结果
- 假设1：不成立，请求正常发送
- 假设2：✅ 成立，密码验证逻辑有bug
- 假设3：不成立，用户数据正常

根因确认：密码验证逻辑bug

→ 进入修复实施

Teammate 2 实施修复...

Lead：修复完成，开始验证

→ 验证通过

✅ Bug 修复完成
```
